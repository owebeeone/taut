# Taut Res+Ext Phase 2 Java Prompt Review - Second Pass

## Findings

### P1 - The Java `Ext` API is still ambiguous enough to produce incompatible implementations

The Phase 1 blockers around missing corpora and scaffold wiring are resolved, but the Java prompt
still leaves the shipped extension API unclear. The base contract names a byte-oriented operation
(`ext_set(host_bytes, tag, ext_value) -> bytes`) and says every language can choose an idiomatic
surface while matching `ext_vectors.json` bytes (`dev-docs/TautResExtP2-Base.md:47`-`64`). The Java
brief sketches ``static byte[] extSet(Cbor host?/byte[] host, long tag, Cbor value)`` and then calls
`Cbor.decode(host)` (`dev-docs/TautResExtP2-Java.md:14`-`17`), which is not a valid signature and
leaves room for a `Cbor` host API, a `byte[]` host API, or overloads.

This matters more now that Phase 1 has wired `Ext.java` into generated runtimes:
`_RUNTIMES["java"]` includes `("Ext.java", "Ext.java")`, and `emit()` will vendor that resource as
soon as the file exists (`src/taut/gen/scaffold.py:32`-`39`, `src/taut/gen/scaffold.py:600`-`607`).
The prompt should pin the Java surface before implementation, for example:

```java
package taut;

public final class Ext {
    public static byte[] extSet(byte[] host, long tag, Cbor value);
    public static Cbor extGet(byte[] host, long tag);
    public static byte[] extClear(byte[] host, long tag);
}
```

It should also name the expected failure mode for below-band tags, preferably
`IllegalArgumentException`, and say whether a non-map top-level `host` is also rejected with
`IllegalArgumentException`. A careful implementer can infer most of this from the base brief, but
the Java prompt as written can still split downstream callers and tests.

### P2 - Package and access assumptions should be explicit for `Ext.java` and the harness

`Cbor.java` is in `package taut`, while `KV` is package-private (`src/taut/gen/runtime/Cbor.java:5`,
`src/taut/gen/runtime/Cbor.java:194`-`198`). Generated Java messages are also package-private, and
their `toCbor()` / `fromCbor()` methods are package-private (`src/taut/gen/java.py:84`-`104`). The
Phase 2 prompt tells the agent to create `src/taut/gen/runtime/Ext.java`, but it does not explicitly
say `package taut;`, `public final class Ext`, or that the Java harness should also declare
`package taut;` if it calls generated `Host` / `Decision` methods.

This is not a semantic blocker, but it is a likely compile-time footgun. An `Ext.java` in the wrong
package cannot construct `KV`, and a harness outside `taut` cannot call `Host.fromCbor`,
`Host.toCbor`, `Decision.fromCbor`, or `Decision.toCbor`. The existing `FloatParity.java` harness
uses `package taut`, so the Java prompt should direct the same pattern.

### P2 - The differential fuzz requirement remains under-specified for Java

The fixed corpus harness is now implementable with JDK stdlib only: `corpus/residual_vectors.json`
and `corpus/ext_vectors.json` are committed, simple hex-row JSON files generated by
`src/taut/corpus/resext_build.py` (`src/taut/corpus/resext_build.py:34`-`78`), and lockstep-tested
against the Python oracle (`src/tests/test_resext_vectors.py:21`-`66`). That resolves the prior
concern that the Java agent would need to invent the missing corpus shape.

The remaining ambiguity is the fuzz part. The base brief asks for "a differential fuzz vs the
Python oracle" with random schemas/values and random extension messages
(`dev-docs/TautResExtP2-Base.md:81`-`83`), while the Java prompt only says "Build a harness over
both corpora + a differential fuzz. JDK stdlib only" (`dev-docs/TautResExtP2-Java.md:21`-`24`).
There is no Java corpus-kit harness generator, no seed/iteration count, and no prescribed way for a
JDK-only Java harness to call or consume a Python oracle for random schemas.

To avoid inconsistent Phase 2 outputs, the prompt should define the minimum acceptable Java fuzz:
for example, a deterministic Python test in `src/tests/test_java.py` that generates random host/ext
cases with the Python oracle, compiles the Java fixture harness, runs those rows through Java, and
reports a mismatch count; or a fixed generated fuzz JSON consumed by the Java harness. Without that,
one agent may do only fixed corpus parity while another attempts a much broader random-schema
generator.

### P2 - Residual ownership still invites unnecessary edits to proven Java encode paths

The current Java residual path is already shaped correctly: generated messages add known fields,
append `wireResidual`, and rely on `Cbor.encode` sorting the combined map
(`src/taut/gen/java.py:88`-`100`, `src/taut/gen/java.py:113`-`115`;
`src/taut/gen/runtime/Cbor.java:147`-`151`). I generated the `ir/resext.taut.py` fixture with
`--lang java --api-only --with-runtime --forward-compat` into `/tmp` and compiled the emitted
`api.java` plus `Cbor.java` with `javac`; it compiled.

The Java prompt still lists `src/taut/gen/runtime/Cbor.java` and `src/taut/gen/java.py` as owned
files (`dev-docs/TautResExtP2-Java.md:6`-`8`). That is technically true if a residual byte mismatch
is found, but the prompt should be more explicit that the residual half is verification-first and
that `Cbor.java` / `java.py` should not be edited unless the corpus produces a real failing byte
case. This reduces regression risk in the FLOAT-proven encode path that the base brief says not to
disturb (`dev-docs/TautResExtP2-Base.md:68`-`74`).

### P3 - Toolchain instructions are stale on this checkout

The Java prompt says PATH `java`/`javac` are broken and requires Android Studio's JDK
(`dev-docs/TautResExtP2-Java.md:21`-`23`). In this checkout, `/usr/bin/java` and `/usr/bin/javac`
both work and report OpenJDK 21.0.10; Android Studio's JDK also exists. This is not blocking, but
the prompt should phrase Android Studio's JDK as a fallback rather than mandatory machine-local
state.

There is a similar local mismatch in the base verification command: `python3 -m pytest` fails here
because `python3` is 3.14 without pytest, while `python -m pytest` and `python run_tests.py` work.
That does not require a Java prompt change unless this environment is the expected runner, but it is
worth noting for the Phase 2 agent.

## Current Implementability Summary

The structural Phase 1 blockers are gone. The Java Phase 2 task is now implementable by a careful
agent who reads the base brief, infers the intended `byte[]` host API, keeps `Ext.java` in
`package taut`, and treats residual work as verification unless the corpus fails.

I would not call the Java prompt fully implementable "as written" for isolated fan-out, because the
main API line is still malformed and the fuzz expectation is not precise enough to produce
consistent test coverage. Tightening those two points should be enough; no additional shared Phase
1 work appears required.
