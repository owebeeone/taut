# Taut Res+Ext Phase 2 Rust Prompt Review 55

## Findings

1. **[Blocker] The Rust Phase 2 prompt is not implementable from this checkout because the Phase 1 oracle artifacts are missing.** `TautResExtP2-Base.md` defines `corpus/residual_vectors.json` and `corpus/ext_vectors.json` as the parse-free truth sources (`dev-docs/TautResExtP2-Base.md:15`-`dev-docs/TautResExtP2-Base.md:22`), and the Rust prompt tells the agent to run both (`dev-docs/TautResExtP2-Rust.md:21`-`dev-docs/TautResExtP2-Rust.md:35`). The plan says those files, the fixture schema(s), and the per-language harness slots are Phase 1 outputs (`dev-docs/TautResExtPlan.md:55`-`dev-docs/TautResExtPlan.md:62`), but the current `corpus/` directory contains only the existing float/glade/griplab artifacts; the committed IR snapshots also have no extension fixtures (`corpus/griplab.ir.json:1236`, `corpus/glade.ir.json:939`). `run_tests.py` only regenerates the existing corpus/build/float/glade/CRDT artifacts (`run_tests.py:17`-`run_tests.py:23`). Until Phase 1 lands, the Rust agent has no vector format, no byte oracle, and no fixture types to generate.

2. **[Blocker] Shipping `ext.rs` through generated runtimes requires a scaffold change that the Phase 2 scope forbids.** The plan explicitly assigns the vendored `ext.<lang>` runtime slot in `_RUNTIMES`/scaffold to Phase 1 (`dev-docs/TautResExtPlan.md:59`-`dev-docs/TautResExtPlan.md:62`), while the base brief says Phase 2 agents must not edit `gen/scaffold.py` (`dev-docs/TautResExtP2-Base.md:63`-`dev-docs/TautResExtP2-Base.md:68`). In the current repo, `_RUNTIMES` emits only Rust `cbor.rs` (`src/taut/gen/scaffold.py:30`-`src/taut/gen/scaffold.py:38`), and `emit(..., runtime=True)` writes only that listed resource (`src/taut/gen/scaffold.py:594`-`src/taut/gen/scaffold.py:599`). Creating `src/taut/gen/runtime/ext.rs` under the Rust-owned files (`dev-docs/TautResExtP2-Rust.md:6`-`dev-docs/TautResExtP2-Rust.md:8`) would not make `tautc gen --with-runtime` copy it, unless the agent violates scope and edits scaffold.

3. **[Major] The cargo/prior-art harness instruction points at an unavailable crate shape.** The Rust prompt says to reuse `razel/crates/razel-wire`'s `vectors.rs` plus `corpus_byte_parity` cargo-test shape (`dev-docs/TautResExtP2-Rust.md:10`-`dev-docs/TautResExtP2-Rust.md:14`) and says `cargo` is available for the harness (`dev-docs/TautResExtP2-Rust.md:34`-`dev-docs/TautResExtP2-Rust.md:35`). This checkout does not contain that crate, and the sibling `trial/rs` tree only has `src/generated.rs` with forward-compat off (`/Users/owebeeone/limbo/taut-dev-cross/trial/rs/src/generated.rs:51`-`/Users/owebeeone/limbo/taut-dev-cross/trial/rs/src/generated.rs:78`). The current in-repo Rust test pattern compiles temporary Rust tests directly with `rustc --test`, not `cargo test` (`src/tests/test_rust.py:41`-`src/tests/test_rust.py:102`). The prompt should either require Phase 1 to add a crate-level Rust harness or tell the Rust agent to use the existing pytest/temp-`rustc` style.

4. **[Major] The prompt is ambiguous about which Rust generator path should produce the forward-compatible fixture.** The Rust generator has the residual implementation behind `_emit_message(..., forward_compat=True)` (`src/taut/gen/rust.py:134`-`src/taut/gen/rust.py:140`, `src/taut/gen/rust.py:153`-`src/taut/gen/rust.py:182`), and scaffold can pass that flag (`src/taut/gen/scaffold.py:231`-`src/taut/gen/scaffold.py:238`). But the direct `rust._emit(schema, golden)` path used by the committed `trial/rs/src/generated.rs` always calls `_emit_message(m)` with the default false flag (`src/taut/gen/rust.py:189`-`src/taut/gen/rust.py:200`), and `test_regen` locks that behavior (`src/tests/test_regen.py:28`-`src/tests/test_regen.py:33`). "Generate the fixture `--forward-compat`" is therefore implementable only through `tautc gen`/`scaffold.rust_api`, not by changing the direct GripLab generator unless regen semantics are intentionally changed. The prompt should state that explicitly to avoid unnecessary generator churn or stale-regeneration failures.

5. **[Major] Extension API and error semantics are underspecified for Rust, which can produce cross-language drift.** The base brief says to mirror `ext.py`, band-check first, and return typed `ExtMsg | null` semantics (`dev-docs/TautResExtP2-Base.md:42`-`dev-docs/TautResExtP2-Base.md:56`), while the Rust prompt gives generic `Cbor` signatures and says the below-band case may be either `panic` or `Err` (`dev-docs/TautResExtP2-Rust.md:28`-`dev-docs/TautResExtP2-Rust.md:32`). Those are different test surfaces. Existing Rust `Cbor` accessors generally panic on misuse (`src/taut/gen/runtime/cbor.rs:20`-`src/taut/gen/runtime/cbor.rs:44`), but choosing `Result<Vec<u8>, _>` would change the signatures in the prompt. The prompt should pin one Rust surface, including whether malformed top-level non-map input panics or returns an error.
