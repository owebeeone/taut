# Taut Res+Ext Phase 2 Rust Prompt Review 55-2

## Findings

1. **[Blocker] Adding the Rust extension runtime will currently make the shared Phase 1 scaffold test fail, so the prompt is not implementable under "keep the suite green" without editing a file the Rust brief does not clearly own.** The Rust prompt assigns the agent a new `src/taut/gen/runtime/ext.rs` (`dev-docs/TautResExtP2-Rust.md:6`-`dev-docs/TautResExtP2-Rust.md:8`), and the base brief says dropping `ext.<lang>` into `gen/runtime/` makes `tautc gen --with-runtime` vendor it (`dev-docs/TautResExtP2-Base.md:24`-`dev-docs/TautResExtP2-Base.md:25`). That is true in the current scaffold: `emit()` vendors every existing runtime resource (`src/taut/gen/scaffold.py:600`-`src/taut/gen/scaffold.py:607`). But `src/tests/test_resext_vectors.py:79`-`src/tests/test_resext_vectors.py:84` still asserts that Rust runtime emission does **not** include `ext.rs`. The first correct Rust implementation that adds the requested file will flip that assertion and fail the full suite. Fix before fan-out by either updating the shared test in Phase 1 to accept/expect `ext.rs` once present, or explicitly granting the Rust agent ownership to update that shared placeholder test.

2. **[Major] The harness/fuzz direction still points at an unavailable Cargo/prior-art layout.** Prior review item 3 is not resolved by Phase 1. The Rust prompt says to reuse `razel/crates/razel-wire`'s `vectors.rs` and `corpus_byte_parity` `cargo test` shape (`dev-docs/TautResExtP2-Rust.md:10`-`dev-docs/TautResExtP2-Rust.md:14`) and then says cargo is available for the harness plus differential fuzz (`dev-docs/TautResExtP2-Rust.md:34`-`dev-docs/TautResExtP2-Rust.md:35`). Cargo is installed, but this checkout has no `Cargo.toml`, `vectors.rs`, or `razel-wire` crate; the sibling `trial/rs` tree contains only `src/generated.rs`, still generated with forward-compat off. The in-repo Rust precedent is the pytest/temp-`rustc --test` harness in `src/tests/test_rust.py:41`-`src/tests/test_rust.py:102`. Since the plan now says `kit.py` harness emission was deferred and agents hand-write harnesses (`dev-docs/TautResExtPlan.md:51`-`dev-docs/TautResExtPlan.md:55`), the Rust prompt should name the expected harness location/shape directly, including what counts as acceptable differential fuzz under "no new deps".

3. **[Major] Rust extension API and error semantics remain under-specified, leaving room for cross-language drift.** The base contract says to mirror `ext.py`, band-check first, and expose typed `ExtMsg | null` semantics for get (`dev-docs/TautResExtP2-Base.md:47`-`dev-docs/TautResExtP2-Base.md:64`). The Rust prompt instead specifies generic `Cbor` signatures and allows either `panic` or `Err` for below-band tags (`dev-docs/TautResExtP2-Rust.md:28`-`dev-docs/TautResExtP2-Rust.md:32`). Python raises `ValueError` below band (`src/taut/ext.py:19`-`src/taut/ext.py:21`), while the existing Rust `Cbor` accessors generally panic on misuse (`src/taut/gen/runtime/cbor.rs:20`-`src/taut/gen/runtime/cbor.rs:84`). The byte corpus will catch happy-path set/get/clear, but it will not pin the public Rust surface or negative behavior. The prompt should choose one Rust surface, for example panic-based `Vec<u8>`/`Option<Cbor>` helpers plus generated-type wrapper usage in tests, or `Result`-returning helpers everywhere.

4. **[Minor] The prompt is now clear enough on the forward-compatible fixture path, but the old direct Rust generator path remains a trap worth naming.** The base brief now says to generate from `ir/resext.taut.py` with `tautc gen` (`dev-docs/TautResExtP2-Base.md:19`-`dev-docs/TautResExtP2-Base.md:20`), and `tautc gen ... --forward-compat --with-runtime` currently emits a Rust `api.rs` with `wire_residual` plus `cbor.rs`. That resolves prior review item 4. The direct `src/taut/gen/rust.py::_emit()` path still calls `_emit_message(m)` without forward compat (`src/taut/gen/rust.py:189`-`src/taut/gen/rust.py:200`), so the prompt should continue steering agents away from changing GripLab regen semantics.
