# Taut Res+Ext Phase 2 Kotlin Prompt Review 55

## Findings

1. **[P0] Phase 2 is not implementable from this worktree because the shared Phase 1 artifacts are absent.**

   The Kotlin brief requires running `residual_vectors.json` and both corpora (`dev-docs/TautResExtP2-Kotlin.md:10`, `dev-docs/TautResExtP2-Kotlin.md:22`), and the base brief names `corpus/residual_vectors.json` plus `corpus/ext_vectors.json` as the byte oracle (`dev-docs/TautResExtP2-Base.md:19`, `dev-docs/TautResExtP2-Base.md:20`). The plan says Phase 1 will emit and commit those corpora, fixture schemas, and harness/runtime slots (`dev-docs/TautResExtPlan.md:55`-`dev-docs/TautResExtPlan.md:62`), but they are not present here. `run_tests.py` only regenerates the existing golden/float/glade/CRDT artifacts (`run_tests.py:18`-`run_tests.py:23`), and `src/taut/corpus/kit.py` still exposes only the Rust harness path (`src/taut/corpus/kit.py:89`-`src/taut/corpus/kit.py:93`). A Kotlin implementer would have to invent fixture schemas, vector formats, and parity harness behavior locally, which defeats the cross-language oracle contract. Fix the prompt by marking Kotlin Phase 2 blocked until Phase 1 lands, or explicitly add the missing shared artifacts and exact row formats to this task.

2. **[P0] The ownership rules forbid the scaffold change needed to ship `ext.kt` with generated Kotlin output.**

   The Kotlin prompt assigns a new `src/taut/gen/runtime/ext.kt` (`dev-docs/TautResExtP2-Kotlin.md:6`-`dev-docs/TautResExtP2-Kotlin.md:8`), while the base brief explicitly says not to edit `gen/scaffold.py` (`dev-docs/TautResExtP2-Base.md:63`-`dev-docs/TautResExtP2-Base.md:68`). In the current repo, `_RUNTIMES` vendors only `cbor.kt` for Kotlin (`src/taut/gen/scaffold.py:30`-`src/taut/gen/scaffold.py:38`), and `emit(..., runtime=True)` writes only that one runtime file (`src/taut/gen/scaffold.py:594`-`src/taut/gen/scaffold.py:599`). As written, a correct `ext.kt` can exist in the package but will not be copied into self-contained generated Kotlin trees, so generated consumers cannot rely on the extension accessor module. Either Phase 1 must add the ext runtime slot before this prompt runs, or the Kotlin prompt must include the scaffold/runtime vending change and its tests in the owned scope.

3. **[P1] The Kotlin harness requirement conflicts with the no-dependency constraint unless the prompt specifies Python-generated vector source.**

   The brief asks for a harness "over both corpora + a differential fuzz" with "JDK stdlib only" (`dev-docs/TautResExtP2-Kotlin.md:20`-`dev-docs/TautResExtP2-Kotlin.md:22`), and the project has no runtime dependencies (`pyproject.toml:21`). Kotlin/JDK stdlib does not provide a JSON parser for the committed corpus files. The existing Kotlin float parity pattern avoids this by hardcoding vector rows in Kotlin source (`src/tests/kotlin_float_parity.kt:3`-`src/tests/kotlin_float_parity.kt:28`) and compiling that source from pytest (`src/tests/test_kotlin.py:107`-`src/tests/test_kotlin.py:125`). Without a specified approach, implementers may add a forbidden dependency, hand-roll a brittle JSON parser, or create a harness that cannot track corpus changes. The prompt should say that pytest loads the JSON corpora, generates a temporary Kotlin vector table/harness, compiles `cbor.kt` + generated API + `ext.kt`, and reports a deterministic fuzz seed plus mismatch count.

4. **[P1] `ext.kt` behavior for non-map hosts is underspecified and can diverge from Python if implemented literally.**

   The base contract says extension accessors operate on top-level host-message maps and mirror `ext.py` (`dev-docs/TautResExtP2-Base.md:42`-`dev-docs/TautResExtP2-Base.md:52`). Python checks the band first, decodes the bytes, and then mutates/reads a dict (`src/taut/ext.py:19`-`src/taut/ext.py:46`); a non-map top-level item fails instead of being rewritten. Kotlin's generic `Cbor` class exposes `mapEntries` as the backing `map` list for every `Cbor`, defaulting to empty for non-map kinds (`src/taut/gen/runtime/cbor.kt:7`-`src/taut/gen/runtime/cbor.kt:14`, `src/taut/gen/runtime/cbor.kt:41`). If an agent follows "decode host, rebuild the pair list" literally (`dev-docs/TautResExtP2-Kotlin.md:14`-`dev-docs/TautResExtP2-Kotlin.md:18`) without a `kind == Cbor.MAP` check, `extSet`/`extClear` can silently convert scalar bytes into maps. The prompt should require an explicit top-level map check and a consistent exception type for below-band tags and non-map hosts.

5. **[P2] The current Kotlin test pattern will skip the available toolchain unless the prompt updates discovery.**

   The brief names the absolute Android Studio `kotlinc` and JBR paths (`dev-docs/TautResExtP2-Kotlin.md:20`-`dev-docs/TautResExtP2-Kotlin.md:21`). In this repo, `src/tests/test_kotlin.py` still gates the compiled harness on `shutil.which("kotlinc")` (`src/tests/test_kotlin.py:107`-`src/tests/test_kotlin.py:111`), so it skips when `kotlinc` is not on `PATH` even though the brief's absolute path exists. The JVM helper is already robust (`src/tests/test_kotlin.py:20`-`src/tests/test_kotlin.py:67`); the prompt should require the same style for `kotlinc` discovery, preferring an env var or the Android Studio path before falling back to `PATH`.

6. **[P2] Kotlin-specific D14 gate coverage is not requested.**

   The base contract requires extension schemas to fail compiled-target generation unless `--forward-compat` is enabled (`dev-docs/TautResExtP2-Base.md:37`-`dev-docs/TautResExtP2-Base.md:40`). The generic scaffold gate currently covers Kotlin because Kotlin is in `_GENERATED` (`src/taut/gen/scaffold.py:574`-`src/taut/gen/scaffold.py:580`), but the existing regression test only exercises Rust (`src/tests/test_forward_compat.py:77`-`src/tests/test_forward_compat.py:81`). Since the Kotlin brief owns `src/tests/test_kotlin.py` (`dev-docs/TautResExtP2-Kotlin.md:8`), it should ask for a Kotlin `scaffold.emit(..., langs=["kotlin"])` assertion that generation fails without `forward_compat=True` and succeeds with it.

7. **[P2] The prompt names `cbor.kt` as owned even though the current residual path appears already green.**

   Kotlin residual re-emission already flows through generated `toCbor` appending `wireResidual` (`src/taut/gen/kotlin.py:110`-`src/taut/gen/kotlin.py:124`), `fromCbor` filtering unknown entries (`src/taut/gen/kotlin.py:138`-`src/taut/gen/kotlin.py:140`), and the runtime map encoder sorting by key (`src/taut/gen/runtime/cbor.kt:151`-`src/taut/gen/runtime/cbor.kt:154`). The base brief says not to touch the proven FLOAT/encode byte path (`dev-docs/TautResExtP2-Base.md:68`-`dev-docs/TautResExtP2-Base.md:69`), but the Kotlin prompt's file list can still encourage unnecessary runtime edits. Narrow the instruction to "verify first; edit `cbor.kt` only for a demonstrated residual/ext accessor need" to reduce float/CBOR regression risk.
